# Problem Solving 方法论(McKinsey 融合版)

源自麦肯锡 Problem Solving 101/102,适配西井场景。用于**开放型命题**——比如"分析港口 AI 调度的市场机会"、"评估仓储场景的 AI 切入路径"。

如果用户已经给了结构清晰的 `.md` 或 outline,直接跳到 SKILL.md 的 Step 3(Dummy Pages 设计),不需要走这份方法论。

---

## STEP 1 · 定义问题边界(Is / Isn't)

目标:明确**研究什么 / 不研究什么**,避免范围蔓延。

### 模板

```markdown
## 问题定义
### 是 ✅(In Scope)
- 核心问题:[一句话]
- 研究范围:[时间 / 地域 / 细分场景]
- 交付物:[PPT 页数 / 风格]
- 受众:[决策层 / 技术评审 / 运营方]

### 不是 ❌(Out of Scope)
- [排除内容 1]
- [排除内容 2]
- [假设已知的内容]
```

### 常见边界问题

- **地域**:国内 vs 全球、华东 vs 长三角
- **时间**:过去 3 年回顾 vs 未来 3 年预测
- **场景**:港口整体 vs 单一码头业务 vs 特定环节(岸桥/场桥/拖车)
- **竞品范围**:直接竞品(同类 AI 调度产品)vs 间接竞品(传统 TOS 厂商)

## STEP 2 · Issue Tree(MECE 拆解)

### MECE 原则

**M**utually **E**xclusive(互斥), **C**ollectively **E**xhaustive(穷尽)。
- 分支之间无交叉
- 所有分支合起来覆盖完整问题

### 结构示意

```
核心问题
├── Lv1 维度 1(独立)
│   ├── Lv2 子问题 1.1
│   └── Lv2 子问题 1.2
├── Lv1 维度 2(独立)
│   └── ...
└── Lv1 维度 3(独立)
```

### 西井常用拆解框架

#### 场景准入分析(机会评估)

```
[场景 X] 是否值得进入?
├── 1. 市场空间(Size)
│   ├── 1.1 场景存量设备规模
│   ├── 1.2 调度痛点的可量化损失
│   └── 1.3 付费意愿与决策链
├── 2. 技术可达性(Feasibility)
│   ├── 2.1 已有技术栈的迁移成本
│   ├── 2.2 数据采集与标注可行性
│   └── 2.3 与场内现有系统的集成路径
└── 3. 商业化路径(Path)
    ├── 3.1 切入点(POC → 小试 → 推广)
    ├── 3.2 合作方式(直销 / 集成商 / 方案打包)
    └── 3.3 首批目标客户画像
```

#### 产品价值论证(方案提报)

```
为什么这个方案能解决客户的问题?
├── 1. 问题的真实性(Problem)
│   ├── 1.1 业务痛点量化
│   ├── 1.2 现有解决方案的局限
│   └── 1.3 未解决将导致的代价
├── 2. 方案的有效性(Solution)
│   ├── 2.1 核心技术机制
│   ├── 2.2 与现有系统的差异化
│   └── 2.3 已验证的场景案例
└── 3. 落地的可行性(Delivery)
    ├── 3.1 实施阶段与时间表
    ├── 3.2 人员与资源投入
    └── 3.3 风险与应对
```

#### 竞争格局分析

```
市场竞争格局如何?
├── 1. 谁在做(Who)
│   ├── 1.1 主要玩家盘点
│   └── 1.2 玩家类型(独立厂商 / 互联网大厂 / 场景方自研)
├── 2. 怎么做(How)
│   ├── 2.1 技术路线差异
│   ├── 2.2 商业模式差异
│   └── 2.3 核心壁垒
└── 3. 往哪走(Where going)
    ├── 3.1 资本与融资动向
    └── 3.2 下一波竞争焦点
```

### MECE 自检清单

```
□ 每个维度是否独立?(无交叉)
□ 所有维度合起来是否完整?(无遗漏)
□ 每个分支都可研究?(有数据来源或内部经验)
□ 层级是否合理?(2–3 层为宜)
□ 与核心问题直接相关?(不跑题)
```

## STEP 3 · Hypotheses(假设驱动)

核心理念:**先假设,再验证**,而不是"先穷举所有数据,再归纳"。后者会浪费 90% 的无效搜索。

### 好假设的三个标准

- **可验证**:能用数据或案例证明 / 证伪
- **具体**:不是"市场在增长",而是"港口 AI 调度渗透率 < 5%,但已有头部玩家 ARR 突破亿元"
- **逻辑自洽**:基于行业常识和初步了解,不是拍脑袋

### 假设形成流程

**1. 快速搜索(5–10 次)—— 只为建立基本认知**

常用搜索模板:
```
"[场景] 数字化 / 智能化 白皮书 2024"
"[场景] AI 调度 市场规模"
"[场景] 头部玩家 融资"
"[场景] 信息化预算"
"[场景] 痛点 / 降本增效"
```

**2. 基于搜索 + 行业经验形成初步假设**

假设模板:
```
H[编号]: [主语] [具体论点], 因此 [结论]
```

西井场景示例:
```
H1: 港口 AI 调度在头部枢纽已规模商用,中小港口是下一波增量
H2: 机场场面调度的技术迁移成本可控,但决策链比港口更长
H3: 仓储场景的核心竞争在与 WMS / TOS 的集成深度,而非算法本身
H4: 客户更在意"决策建议 + 人工保留终审权",纯自动化接受度低
```

**3. 把假设挂到 Issue Tree 对应节点**

```
Issue Tree 节点 → 对应假设 → 验证所需证据
1.1 场景存量 → H1(渗透率 < 5%) → 行业报告 / 海关数据
2.1 技术可达性 → H2(迁移成本可控) → 内部 PoC 数据 / 友商案例
```

### 假设与叙事的关系

Phase 1 输出的 Hypotheses 会直接变成 PPT 的**章节结构**和**每页的论点式标题**(见 SKILL.md 的 Consulting Storyline Principles 部分)。每个假设在 PPT 里对应的都是 1 页或 1 组页面——证据就是这一页 / 这一组的 body。

## 方法论要点(对比)

### 假设驱动 vs 数据驱动

| 维度 | 假设驱动(McKinsey) | 数据驱动(传统) |
|------|----------------------|----------------|
| 起点 | 先有框架和假设 | 先收集所有数据 |
| 效率 | 高,精准搜索 | 低,大量无效工作 |
| 质量 | 高,有明确论点 | 中,可能缺乏洞察 |
| 适合 | 复杂商业命题 | 指标驱动的决策 |

### MECE vs 思维导图

| 维度 | MECE Issue Tree | 普通思维导图 |
|------|-----------------|-------------|
| 结构 | 严格不重不漏 | 随意联想 |
| 可研究性 | 每节点可量化 | 部分难以验证 |
| 逻辑性 | 层层递进 | 平铺展开 |
| 适合 | 商业命题分析 | 创意发散 |

### Storyline 构建(Phase 1 的产物 → Phase 2 的 Dummy)

```
开场:   问题定义 + 核心结论(执行摘要)
  ↓
第一章: 场景现状(H1–H2 验证)
  ↓
第二章: 驱动因素(H3–H4 验证)
  ↓
第三章: 竞争格局(H5–H6 验证)
  ↓
结尾:   机会 / 建议(综合所有 H)
```

**Storyline 自检**:
```
□ 每页标题都是完整论点(不是 topic label)?
□ 相邻页面有逻辑连接?
□ 整体讲了一个完整的故事?
□ 结论自然从证据得出?
```

## 何时用、何时不用

**用**:
- "帮我分析 X 市场 / 场景 / 机会"
- "给 X 做一份战略 / 方案 / 白皮书"
- 用户只有问题和资料,还没有清晰结构

**不用**:
- 用户已经给了 outline 或 structured `.md` → 直接到 Step 3(Dummy Pages)
- 用户只是要"把这份材料做成 PPT" → 跳过 Phase 1
- 场景是产品介绍 / 定期汇报 → 结构通常已有套路,不需要重新 MECE
