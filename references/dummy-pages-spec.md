# Dummy Pages 设计规范(依赖感知)

Dummy Pages 是"先把 PPT 的**每一页**设计清楚,再动手写代码"的产物。输出是一个 `.md` 文件,**落盘保存**,支持跨对话续写。

---

## 为什么要 Dummy

- **减少返工**:结构错了比某一页丑代价大得多,先对齐结构
- **人机共识**:用户在 `.md` 上拍板一次,后面的逐页生成不需要反复确认方向
- **跨对话续写**:新对话里上传 Dummy + 已生成的 `.pptx`,Claude 可以从第 N 页继续
- **依赖显式化**:每一页写清楚"依赖谁""生成顺序",避免在新对话里生成执行摘要时发现没素材

---

## Dummy.md 文件命名

```
<项目名>_DummyPages_<YYYYMMDD>.md
```

例:
- `ATC_DummyPages_20260415.md`
- `LOSA_港口方案_DummyPages_20260415.md`

建议放在跟最终 `.pptx` 同一个目录,便于查找和归档。

---

## 依赖类型

5 种标签,按页面生成的**独立性**从强到弱排列:

| 标签 | 含义 | 典型例子 |
|------|------|---------|
| ✅ **独立** | 无依赖,任何时候可生成 | 封面、章节分隔、单页数据分析 |
| ⏩ **依赖前页** | 需要先完成某几页 | 对比页(需要基础数据页)、深入分析页 |
| ⏪ **依赖后页** | 需要后面页面完成后才能写 | 章节总结页、执行摘要(理想做法) |
| 📄 **需要文档** | 依赖 Issue Tree / 内部资料等外部文档 | 执行摘要(需 Hypothesis Tree)、行业对标页 |
| ⏸️ **最后生成** | 前后都依赖,务必最后写 | 全局目录、最终结论页 |

## Dummy.md 结构模板

```markdown
# <项目名> Dummy Pages

## 项目信息
- 创建日期: YYYY-MM-DD
- 总页数: XX 页
- 预计章节: X 章
- 受众: [决策层 / 技术评审 / 运营方]
- 叙事模式: SCQA / BCG / Bain(见 SKILL.md)
- 核心结论:[一句话]

## 视觉规范
见 `references/design-system.md` 和 `references/layouts-guide.md`。本文件不复制规范,
只负责结构、内容、依赖关系。

## 页面依赖关系总览

建议生成顺序:

**第一轮:独立页面**(可任意顺序)
- 第 1 页 (封面) ✅
- 第 4–9 页 (基础分析) ✅

**第二轮:前向依赖页面**(依赖第一轮)
- 第 10 页 (对比 / 深入) ⏩ 需要第 4–9 页

**第三轮:后向依赖页面**(最后生成)
- 第 2 页 (执行摘要) ⏪ 需要正文所有页
- 第 3 页 (目录)     ⏸️ 所有页面完成后生成

1. 上传本 Dummy 文件
2. 上传已生成的 `.pptx`(可选,便于 Claude 看到当前进度)
3. 告诉 Claude:"从第 N 页继续"
```

## Claude 设计 Dummy 时的要求

1. **每页必须有**:依赖关系、slide type、内容要点、数据需求
2. **不是独立页的,必须有**:前置条件、依赖页面、缺失时对策
3. **Dummy 顶部必须有**:项目信息、依赖关系总览(含建议生成顺序)、断点续写说明
4. **页面标题必须是论点**,不是 topic label(见 SKILL.md 的 Consulting Storyline Principles)
5. **Slide type 必须能映射到 builder 实际方法**(参考 `layouts-guide.md` 和 `layouts-analytic.md`),分析型如需外部图要说明"PNG + image"
6. **信息来源字段写清楚**:是 web_search 还是内部资料,以及具体关键词或文档名

## 跨对话续写流程

新对话中用户上传 Dummy 后,Claude 要:

1. `Read` 整份 Dummy,记住:总页数、章节划分、依赖关系总览
2. 查看用户要生成的页面在哪一轮
3. 如果是独立页 → 直接跳到该页的 Slide type 开始生成
4. 如果有依赖 → 先问:"你已经生成到第几页了?把 `.pptx` 发我一下好定位"
5. 根据当前进度提示用户走哪条路径(第一轮 / 第二轮 / 第三轮)
6. 从该页开始进入逐页循环(见 `data-collection.md`)
